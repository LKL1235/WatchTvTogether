# 一起看（Watch Party）

基于 Go 后端 + Vue 前端的同步观影应用规划。

## Todo / 开发计划

### 1. Go Web 服务基础设施

- [ ] 使用 Go 搭建 HTTP API 服务（路由、中间件、健康检查、配置与环境变量）。
- [ ] 数据库/会话存储选型（如 PostgreSQL、Redis：用户与 OAuth 会话、房间、授权缓存）。

### 1.1 用户与 OAuth 鉴权（角色与权限）

- [ ] 接入 OAuth2/OIDC 提供商，完成登录回调与 token 换用户信息。
- [ ] 用户模型：**管理员**（全局管理：用户、系统配置、媒体库、任务与审计）与**普通用户**（业务功能）。
- [ ] 基于角色与房间角色的授权中间件：管理员绕过房间限制时保留审计需求（可选）。

### 1.2 Vue 前端：房间与同步观看

- [ ] 房间：创建/加入/邀请链接或邀请码；**房主**拥有播放与队列管理权限；**同房间观众**只读同步、无控播权。
- [ ] 播放能力：支持 **m3u8、mp4 及常见流式/网络地址**；播放器与后端进度协议对齐（见 1.3）。
- [ ] **房主导控**：播放/暂停/seek/切换当前视频/顺序或指针移动。
- [ ] **视频队列**：房主可追加 URL（第三方源）或从**服务器已有资源**选择；非房主仅展示与被动同步。

### 1.3 视频进度与状态：接口形态与方案比选

- [ ] **服务端**：维护「房间级」统一状态（`currentItem`、`position`、`isPlaying`、`updatedAt` 等）及变更入口（仅房主或鉴权后写入）。
- [ ] **推荐方案**（在多数“一起看”场景下更稳妥）  
  - **主推荐：WebSocket（双向、低延迟）**  
    - **优点**：单向广播进度与指令延迟低、易做 ACK/重连、可叠加聊天与在线列表。  
    - **缺点**：需处理连接与心跳、在部分企业网络需兼容代理；运维略重于纯 HTTP。  
  - **替代：Server-Sent Events (SSE)（仅服务端推）**  
    - **优点**：实现简单、基于标准 HTTP/长连接、对“只下发型”状态广播足够。  
    - **缺点**：**不适合房主频繁 seek** 的上行反馈时仅靠 SSE 会别扭（常需再配一个 HTTP POST/PUT 上报，形成双通道）；对双向实时性弱于 WebSocket。  
  - **纯轮询 HTTP**  
    - **优点**：极简单。  
    - **缺点**：延迟大、费带宽，不适合高频率进度同步。  
- [ ] **结论与落地**：以 **WebSocket 为主** 做进度/队列/房态广播；**REST** 做房间/媒体/鉴权/配置；**如需极简部署** 再评估 **SSE + 房主操作走 REST** 的折中，但需接受略高的架构割裂感。

### 1.4 管理端：下载/入库接口

- [ ] 在**管理端**仅管理员可调用的接口：根据 URL 将资源拉取并**落盘/入库**为服务端可播放资产。
- [ ] 支持类型：**m3u8（HLS 拉取/合并或保留清单+分片）**、**直链 mp4/常见直传格式**、**其他常见网络地址**、**磁链/种子**（可对接 aria2、qBittorrent 等，异步任务+回调状态）。
- [ ] 防滥用：大小限制、白名单/黑名单、任务队列、失败重试与可观测性。

### 1.5 资源查询与海报

- [ ] **查询 API**：列表/分页/标签；返回**名称、时长、**以及**海报图 URL**（或内联生成策略说明）。
- [ ] **海报生成**：用 ffmpeg 对本地缓存文件**抽取关键帧**作为封面；异步任务、持久化到对象存储或本地 `posters/{id}.jpg`。
- [ ] 与 1.2/1.3 一致：仅展示服务端已**缓存/入库**的项；权限按角色与房间需要裁剪。
