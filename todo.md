# WatchTogether 需求 Todo

## 注意事项与约束

- [ ] 仅在需求明确后再拆分实现任务：本文件同时覆盖 Go 服务端与 Web 客户端需求，但当前仓库是后端仓库；客户端事项需要同步到对应前端仓库或前后端联调任务中。
- [ ] 保持前后端接口契约一致：新增或调整接口时需要同步更新请求/响应字段、错误码、鉴权要求、房间权限校验规则，以及客户端调用时机。
- [ ] Redis 数据需要设计统一 key 规范、TTL 与清理策略，避免房间、用户在线状态、队列和播放进度之间出现脏数据或孤儿 key。
- [ ] 房间状态应以 Redis 中的实时状态为准，Postgres 仅保留需要持久化的用户、视频、历史记录或审计类数据；迁移时需要明确哪些字段不再从数据库读取。
- [ ] 多端登录允许同一用户存在多个会话/token，但房间成员关系必须按用户维度唯一，一个用户在任意时刻只能属于一个房间。
- [ ] 房主权限必须在服务端强校验，客户端禁用控件只作为体验优化，不能替代后端对播放、暂停、进度同步、队列变更等操作的权限判断。
- [ ] 播放状态同步需要考虑时钟差异、网络延迟和乱序请求；所有状态写入应携带服务端记录时间，并在必要时拒绝旧版本事件覆盖新状态。
- [ ] 房间关闭、自动清理、用户离开、断线重连、重新进入房间等流程需要统一处理成员状态、播放状态、队列状态和实时通知。
- [ ] Ably 实时消息仍应作为客户端同步通知通道；Redis 负责保存可恢复状态，避免用户刷新或重新进入后只能依赖实时消息历史。
- [ ] 每个需求落地时需要补充自动化测试或联调验证，至少覆盖权限、并发、多端登录、空房间清理、播放进度恢复和队列同步等关键路径。

## 服务端 todo

- [ ] 将房间当前用户迁移到 Redis 存储：为每个房间维护成员集合，并记录用户与房间的反向索引；加入房间、离开房间、踢人、关闭房间和清理任务都必须同步更新这两类 key。
- [ ] 将房间内视频队列迁移到 Redis 存储：使用有序结构保存队列顺序、当前播放项、播放模式和必要的视频元数据引用；队列新增、删除、重排、切换视频后需要发布实时同步消息。
- [ ] 将房间播放进度迁移到 Redis 存储：保存房间 id、当前视频 id、播放进度、播放状态、服务端更新时间戳和最近一次控制事件版本，用于用户重新进入时恢复当前状态。
- [ ] 减少房间实时状态对数据库的依赖：梳理创建房间、获取房间列表、获取房间详情、加入房间、获取房间 state/snapshot/control 等接口，改为优先读取 Redis 中的成员、队列和播放状态。
- [ ] 新增或完善关闭房间接口：为房主或具备权限的用户提供关闭房间能力，关闭时删除 Redis 房间成员、用户房间索引、视频队列、播放进度等相关 key，并通过 Ably 通知房间内客户端退出或刷新。
- [ ] 在用户登录流程中触发全局自动清理检查：从 Redis 读取全局 `last_room_cleanup_at` 时间戳；如果距离当前时间超过 5 分钟，则尝试获取分布式锁并执行清理，避免多个登录请求并发重复清理。
- [ ] 实现空房间自动关闭逻辑：清理任务扫描 Redis 中的活跃房间或房间成员集合，关闭成员数为 0 的房间，并确保清理数据库残留状态、Redis key 和实时订阅通知的一致性。
- [ ] 支持多端同时登录：登录接口不应因为同一用户已有有效 token 或在线会话而拒绝新登录；会话刷新、退出登录和 token 校验逻辑需要保持多端兼容。
- [ ] 限制单个用户只能出现在一个房间：加入房间时检查 Redis 用户房间索引；如果用户已在其他房间，应拒绝加入、提示先离开原房间，或按产品决策自动从旧房间移除后再加入新房间。
- [ ] 在播放、暂停、同步进度接口中记录播放状态快照：每次房主调用 control/snapshot 类接口时，将房间 id、当前视频 id、播放进度、播放状态和服务端时间戳写入 Redis。
- [ ] 支持根据 Redis 快照计算重新进入时的当前进度：用户进入房间时读取播放状态快照；如果状态为播放中，则使用当前服务端时间减去快照时间推算实时进度，如果为暂停则直接返回快照进度。
- [ ] 处理播放进度边界条件：推算进度不能小于 0，不能超过当前视频时长；超过视频时长时需要结合队列播放模式决定是否停留、切到下一个视频或循环播放。
- [ ] 为队列与播放状态设计 Redis key 与 TTL：例如房间成员、用户所在房间、房间队列、房间播放状态、全局清理时间、清理锁等 key；TTL 需要与房间生命周期和自动清理策略一致。
- [ ] 补充服务端测试：覆盖 Redis 状态读写、加入房间互斥、多端登录、关闭房间、登录触发清理、空房间清理、播放状态恢复和房主权限校验。

## 客户端 todo

- [ ] 优化视频队列提交流程：用户点击添加视频时立即调用服务端队列接口提交新增项，并通过实时消息让房间内所有用户立刻看到最新队列，而不是等切换视频时才提交。
- [ ] 在队列 UI 中展示服务端权威状态：客户端本地队列只能作为乐观更新或加载态，最终以服务端返回和 Ably 推送的队列顺序、当前播放项、播放模式为准。
- [ ] 大厅添加刷新按钮：在房间列表页面提供手动刷新入口，点击后重新请求服务端房间列表，并处理加载中、请求失败、空列表和刷新后的房间状态变化。
- [ ] 改为监听视频播放器原生操作：房主在播放器上执行播放、暂停、拖动进度、跳转等操作时，客户端应直接调用后端控制接口同步状态，不再依赖额外的播放/暂停/同步进度按钮。
- [ ] 移除视频下方额外进度 UI：删除“进度”文字、额外进度条和与其绑定的手动同步交互，避免与播放器原生控制栏产生重复或状态不一致。
- [ ] 将房主播放器操作同步给其他用户：房主触发播放、暂停或进度变化后，客户端调用后端接口，由服务端写入 Redis 并通过 Ably 广播；其他客户端收到消息后更新播放器状态。
- [ ] 避免播放器事件回环：非房主客户端因收到同步消息而更新播放器时，需要区分远端同步与用户本地操作，避免再次调用控制接口造成循环广播。
- [ ] 队列添加顺序播放模式：当前视频播放结束后，客户端根据服务端队列状态进入下一个视频；如果已经是最后一个视频，则按产品规则停止播放或停留在末尾。
- [ ] 队列添加循环播放模式：支持循环播放当前队列；当前视频结束后自动进入下一个视频，最后一个视频结束后回到队列第一个视频。
- [ ] 将播放模式同步到服务端：顺序播放/循环播放的切换应调用服务端接口保存到 Redis，并广播给房间内所有客户端，保证刷新和重新进入后仍保持一致。
- [ ] 监听视频播放结束事件：播放器触发 ended 时，如果当前用户是房主，则根据服务端队列和播放模式请求切换到下一个视频；非房主只等待服务端/Ably 同步，不主动推进队列。
- [ ] 禁用非房主播放器控制栏：非房主进入房间时播放器应处于只读观看状态，禁止播放、暂停、拖动进度、倍速、切换视频等会改变房间状态的控制行为。
- [ ] 保留非房主必要观看能力：禁用控制栏时仍需评估是否允许全屏、音量、本地静音、画质等不影响房间共享状态的本地操作，并在 UI 上清晰区分可用与不可用控件。
- [ ] 进入房间时恢复播放状态：客户端进入或刷新房间后调用服务端状态接口，使用服务端返回的当前视频、队列、播放模式、播放状态和计算后的进度初始化播放器。
- [ ] 补充客户端联调测试：覆盖添加队列即时同步、大厅刷新、房主原生播放器控制同步、非房主禁止控制、顺序播放、循环播放、刷新后恢复进度和多用户同房间观看场景。
